Terraform Quickstart

17 March 2022·1663 words·8 mins
Author
Jesse Molina
Engineer. Developer. Researcher.

Terraform is an open-source tool that allows cloud provider APIs to be expressed as declarative configuration files, enabling Infrastructure-as-Code.

In layman's terms, rather than manually creating resources on Azure, AWS, GCP, etc., we can express our resource architecture as code (hence IaC).

With our cloud infrastructure defined as code, we are able to take advantage of DevOps tools and practices to maintain, test, and deploy our configuration via CI/CD pipelines. Furthermore, it facilitates our ability to scale our services in the cloud and allows engineers to have a better mental model of our deployments.

In this post, we will review the main concepts of Terraform in order to get you started. We will use Azure in our examples, but the same logic applies for any other provider.

Terraform Setup

Visit the Terraform installation page to get started.

In my case, I can use a package manager to search for and install Terraform.

pacman -Ss terraform
community/terraform 1.1.6-1 [installed]
    HashiCorp tool for building and updating infrastructure as code idempotently

Once installed, we can validate the terraform version in use.

terraform version
Terraform v1.1.6
on linux_amd64

A terraform project can be started within an empty.

mkdir example
cd example

File Extensions

Terraform uses file extensions to define a file's functionality and purpose within a Terraform project. In fact, filenames default to terraform.tfextension.

example/
├── terraform.tf
├── terraform.tfplan
├── terraform.tfstate
├── terraform.tfvars

.tf

The primary file extension for Terraform code. This file defines what cloud provider to use and what resources will be created on it. It is common practice to rename this file as main.tf and for code to be split up into multiple .tf files to separate the logic in the code.

.tfvars / .tfvars.json

Defines the variable values that are passed into the Terraform code. These values allow us to dynamically define and build the Terraform environments. It is possible to have multiple .tfvars files, for example, on the basis of environments (i.e dev.tfvars, prod.tfvars).

Variables can also be written in JSON using the .tfvars.json file extension.

.tfstate

A generated state file that keeps track of all resources in a Terraform environment. While this file can exist within the source code, it's often best to store the state in a remote location when collaborating with others.

.tfplan

A generated execution plan that defines the actions Terraform would take if the configurations were to be applied. Plans can be saved and passed into Terraform to apply.

Configuration Syntax

Terraform code is written in Blocks. There are many different blocks, each with their own functionality and arguments.

In example, a resource block is used to create an azure resource group named example with a value provided for the name argument.

resource "azurerm_resourcegroup" "example" {
  name = "my-example-resource-group"
  ...
}

The following are the different blocks that can be defined in the code.

example/
├── main.tf

Providers

Providers are Terraform plugins that allow Terraform to work with cloud providers and APIs.

There are many open source providers to choose from, each with their unique configurations and documentation; check out the terraform registry to browse the available options.

In this example, we are provisioning Azure resources using the azurerm provider and specify the required version.

terraform {
  required_providers {
    azurerm = {
      source = "hashicorp/azurerm"
      version = "=2.91.0"
    }
  }
}

We can expand on our provider block by changing the default backend for where the .tfstate file will be hosted.

In this example, we are using an Azure Blob Storage container to host our Terraform state via "azurerm". Alternative backends are HashiCorp Vault or most cloud object storage providers (i.e. AWS S3, GCP Cloud Storage).

terraform {
  required_providers {
    ...
  }
  backend "azurerm" {
    resource_group_name  = "example-resource-group"
    storage_account_name = "examplestorageaccountname"
    container_name       = "example-containter-name"
    key                  = "terraform.tfstate"
  }
}

A connection to our Azure subscription can be initialized with a provider block for "azurerm". Rather than hard coding values into the argument block, variables from definitions can be passed instead.

provider "azurerm" {
  subscription_id = "example-subscription-ido"
  tenant_id       = var.sub.tenant_id
  features {}
}

Resource

Resource blocks are the primary blocks used in Terraform code. They are used to create and describe one or more infrastructure objects, such as kubernetes clusters, application gateways, etc.

Each provider exposes their corresponding resources, all of which have their unique definitions. Be sure to review the documentation for whatever provider you intend to use.

resource "azurerm_resourcegroup" "example" {
  name     = "rg-example"
  location = "West Europpe"
}

Variables

Variable blocks define the variables that can be dynamically passed into the terraform code.

Variables are defined with the name, type, and optionally, the default value. The different types of variables include string, bool, number, and map.

variable name {
  type = string
}

variable count {
  type = number
  default = 2
}

More complex variables can be defined as type object, whose values can be accessed using dot notation.

variable sub {
  type = object {
    subscription_id = string
    tenant_id       = string
  }
}

Data

Data blocks are used to gather information from existing resources.

data "azurerm_subnet" "example" {
  name = var.sn_name
  virtual_network_name = data.azurerm_virtual_network.default.name
  resource_group_name = data.azurerm_virtual_network.default.resource_group_name
}

These code blocks allow us to pass values into other code blocks using dot notation. For example, we can pass a data.azurerm_subnet.default.id value into the subnet_id argument, as shown below.

resource "azurerm_network_interface" "example" {
  ...

  ip_configuration {
    name                          = "internal"
    subnet_id                     = data.azurerm_subnet.example.id
    private_ip_address_allocation = "Dynamic"
  }
}

Modules

All Terraform providers can be considered to be modules: collection of Terraform code that facilitates our ability to create resources.

In similar fashion, we can create our own Terraform modules to ease code repetition and abstract collection of resources we might want to build in conjunction (i.e. network load balancer with collection of load balancer rules).

This post won't dive into modules, but in essence, modules are written in Terraform code and can be saved in their own respective directory (i.e. modules). In the modules block, the directory location is passed into the source argument.

module "load_balancer_west_europe" {
  source = "modules/app_load_balancer"
  name = var.app_weu.load_balancer_name
  ...
}

module "load_balancer_east_europe" {
  source = "modules/app_load_balancer"
  name = var.app_weu.load_balancer_name
  ...
}

Visit the modules documentation for more details.

Outputs

Output blocks allow us to export resources or resource properties to be used within our Terraform code.

The most common use case for an Output is for when we write modules. For example, our module can create a load balancer whose load balancer ID may be leveraged to build other resources.

We don't necessarily have to expose everything that is created within our module, but instead can share specific parameters.

output "load_balancer_id" {
  value = azurerm_lb.example.id
}

Again, we can leverage the output parameters for other resources. In this example, the load_balancer_east_europe module has exposed the load_balancer_id.

resource "azurerm_lb_backend_address_pool" "example" {
  name                = "bep-example"
  loadbalancer_id     = module.load_balancer_east_europe.load_balancer_id
  resource_group_name = "rg-example"
}

File Structure

While we can write out entire Terraform code into one file, it's often best to break the code into multiple files. As stated earlier, Terraform doesn't care for the name of the file, but the extension that is being used, giving us some liberty for how we want to name our files. While there isn't a definitive naming convention, it is common to write the code blocks into their respective file.

example/
├── main.tf
├── outputs.tf
├── variables.tf
├── terraform.tfvars
├── ...

The file structure can be more complex as well, especially when we consider modules and code for multiple environments.

For example, I tend to write my terraform code as follows. There is no standard convention for directory structure either, so you're free to build your project as you see fit.

├── environments
│   └── dev
│       ├── dev.tfvars
│       ├── main.tf
│       └── variables.tf
│   └── prod
│       ├── prod.tfvars
│       ├── main.tf
│       └── variables.tf
├── modules
│   ├── app_load_balancer
│   │   ├── main.tf
│   │   ├── outputs.tf
│   │   └── variables.tf
│   ├── app_vm_cluster
│   │   ├── main.tf
│   │   └── variables.tf
├── plans
│   └── dev.tfplan
│   └── prod.tfplan

CLI Commands

There are three primary command-line arguments used when developing Terraform code.

terraform init

Terraform init initializes the backend and installs all of our dependencies. It's often best practice to run this command whenever you make significant code changes.

terraform init
Initializing the backend...

Initializing provider plugins...
- Reusing previous version of hashicorp/azurerm from the dependency lock file
- Using previously-installed hashicorp/azurerm v2.91.0

Terraform has been successfully initialized!

You may now begin working with Terraform. Try running "terraform plan" to see
any changes that are required for your infrastructure. All Terraform commands
should now work.

If you ever set or change modules or backend configuration for Terraform,
rerun this command to reinitialize your working directory. If you forget, other
commands will detect it and remind you to do so if necessary.

terraform plan

Terraform plan is the precursor to actually applying any changes and allows you to review what changes will be applied once executed.

terraform plan

The CLI output will show all new, modified, and deleted resources and properties.

Terraform detected the following changes made outside of Terraform since the last "terraform apply":

  # module.cribl_leader_vm_eus2.azurerm_network_interface.cribl_leader has changed
  ~ resource "azurerm_network_interface" "app_vm_cluster" {
        id                            = "/subscriptions/fake-subscription/fake-object-id"
        name                          = "example-name"
      + tags                          = {
            "Cost_Center_ID"   = "111000"
            "Customer"         = "Internal"
            "Environment_Type" = "Dev"
            "Product_Group"    = "Internal"
        }
        # (10 unchanged attributes hidden)

        # (1 unchanged block hidden)
    }

Upon running, Terraform will generate a terraform.plan file. You can create many plans, which can then be passed into Terraform to apply later.

terraform apply

Once we're satisfied with our Terraform code, we can put it into action by running terraform apply.

By default, you will be prompted to confirm the changes before applying.

terraform apply

A word of caution! I advise that you run Terraform plan prior to running apply to verify what the changes will be. It's completely possible to accidentally delete or modify a resource without intention.
